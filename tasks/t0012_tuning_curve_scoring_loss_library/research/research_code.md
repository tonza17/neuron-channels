# Research: code for t0012 tuning-curve scoring/loss library

## Task Objective

Build a pure-Python library that scores any candidate tuning curve (12-angle firing-rate vector,
optionally with per-trial repeats) against the canonical target from [t0004] using a weighted
combination of DSI, peak-at-preferred, null-residual, and HWHM components, plus a Poleg-Polsky
envelope penalty. The identity test `score(target, target) == 0.0` pins the library's formulas to
whatever [t0004] produced, so code research must pin the exact DSI and HWHM conventions used there
before any implementation.

## Library Landscape

No libraries are yet registered under any task's `assets/library/`. The project's tasks 1-14 have

## Key Findings

### DSI formula to mirror

[t0004] `code/generate_target.py::compute_dsi` computes `DSI = (pref - null) / (pref + null)` with
`pref = r_base + amplitude * (1.0 ** n) = r_peak` and
`null = r_base + amplitude * (0.0 ** n) = r_base`. For the default params (`r_base=2`, `r_peak=32`,
`n=2`) that gives `DSI = 30/34 ≈ 0.8824`, which is in the acceptance band `DSI_MIN=0.6`,
`DSI_MAX=0.9` hard-coded at `generate_target.py:53-54`. On an empirical 12-angle curve, `pref` and
`null` are the rates at the preferred angle and the opposite angle — not arbitrary max and min —
and the DSI function the scoring library exposes must reproduce exactly this definition so that
`score(target_csv, target_csv) == 0.0` holds.

### Target dataset schema from [t0004]

The canonical dataset at `tasks/t0004.../assets/dataset/target-tuning-curve/files/` is three
CSV/JSON files:

* `curve_mean.csv` with columns `angle_deg` and `mean_rate_hz`, 12 rows at 0°, 30°, …, 330°
  (generated by `np.linspace(0, 360, 12, endpoint=False)` at `generate_target.py:207-212`)
* `curve_trials.csv` with columns `angle_deg`, `trial_index`, `rate_hz`, 240 rows (12 angles × 20
  trials)
* `generator_params.json` mirroring the `GeneratorParams` dataclass

The scoring library must therefore accept **either** a 12-row mean CSV **or** a 240-row trials CSV;
when trials are supplied it must fold them into a per-angle mean before the DSI / peak / null / HWHM
computations and additionally emit a reliability score (e.g., per-angle CV or split-half
correlation).

### HWHM convention

[t0004] does not compute HWHM directly — it emits the 12 closed-form samples and relies on
downstream consumers. The closed-form tuning curve is
`r(theta) = r_base + (r_peak - r_base) * ((1 + cos(theta - theta_pref)) / 2) ** n` at
`generate_target.py:65-67`. HWHM is the angular distance from `theta_pref` at which
`r(theta) = (r_peak + r_base) / 2`. For default params (n=2) this resolves to ≈ 65.5°
analytically (`((1+cos(phi))/2)^2 = 0.5` → `phi = arccos(sqrt(2)-1) ≈ 65.5°`). On a 12-angle
empirical curve the library must resolve HWHM by linearly interpolating between the sampled points
on the decreasing side (from `theta_pref` toward the null) and inverting the interpolant at
`(r_peak + r_base)/2`, with `scipy.interpolate.interp1d` or `numpy.interp`. The interpolation must
be monotone on each side of `theta_pref` so that `interp1d(..., kind="linear")` returns a single
root.

### Numerical stability and angle handling

Angles are stored in degrees modulo 360, but trigonometry in [t0004] is in radians (`np.deg2rad` at
`generate_target.py:64, 135`). The loss library must keep angles in degrees at the API boundary (so
loaded CSVs work directly) and convert to radians only inside the closed-form helpers. The null
direction is `(theta_pref + 180) mod 360`, not `(theta_pref - 180)` — the expression
`r_base + amplitude * (0.0 ** n)` at `generate_target.py:74` evaluates to `r_base` regardless of
sign, so both conventions agree, but the scoring API must pick the modulo form to avoid negative
angles feeding into `np.interp` which requires sorted `xp`.

### Trial-level reliability pattern

[t0004] `sample_trials` produces rates by adding Gaussian noise with `noise_sd_hz=3.0` and clipping
at zero (`generate_target.py:78-90`). Reliability across 20 trials at each angle can therefore be
modelled as either (a) coefficient of variation `std/mean` over trials, or (b) split-half Pearson
correlation of trial means. The project's published convention has not been chosen; the library
should expose both and default to split-half Pearson, since CV is undefined near zero rates and the
null-direction rate in the target is only ~2 Hz.

## Lessons Learned

* [t0004] chose `np.linspace(0, 360, n_angles, endpoint=False)` for angles, not
  `np.linspace(0, 2*pi, n_angles)`. Keeping degrees at the boundary and converting internally avoids
  floating-point drift around 360° wrap-around.
* [t0004] clips sampled rates at zero (`np.clip(..., a_min=0.0)` at `generate_target.py:90`). The
  scoring library must assume non-negative rates and raise a clear error if a candidate has negative
  entries.
* [t0004] validates `DSI_MIN <= dsi <= DSI_MAX` via `assert` at `generate_target.py:202-204` and
  exits non-zero if the band is violated. The scoring library should not abort on out-of-band
  candidates — only report the DSI — because scoring candidate fits that are deliberately worse
  than the target is the whole point.
* [t0009] (calibrate dendritic diameters) demonstrated that "replace placeholder field X with
  literature-derived value" tasks benefit from a clear identity test (new value must differ from
  placeholder at specified positions). The analogue here is the `score(target, target) == 0.0`
  identity test, which gives us a hard anchor for every future CI run.
* [t0007] (install NEURON+NetPyNE) is a read-only dependency for downstream simulation tasks but is
  not needed by the scoring library; the library is pure Python+NumPy+SciPy and must not import
  NEURON, so the pytest suite runs without the paid simulator toolchain.

## Recommendations for This Task

1. **Mirror [t0004] DSI exactly** — implement
   `compute_dsi_from_curve(angles, rates, theta_pref) = (rate_at_pref - rate_at_null) / (rate_at_pref + rate_at_null)`
   and add a unit test that loads `curve_mean.csv` and asserts the result matches
   `generator_params.json` to within 1e-9.
2. **Use `np.interp` (not custom root-finding) for HWHM** — restrict to the 6 angles on the
   decreasing side from `theta_pref` to `theta_pref + 180°`, sort by angle, and call `np.interp`
   inverting on rate. Assert the interpolant crosses the half-max exactly once per side.
3. **Copy [t0004]'s `paths.py` layout** into `tasks/t0012.../code/paths.py` and expose the default
   target dataset at a named constant (e.g., `DEFAULT_TARGET_MEAN_CSV`).
4. **Expose both mean-curve and trials inputs** at the API boundary —
   `score(candidate_mean, target_mean=...)` and `score(candidate_trials, target_trials=...)` — so
   downstream optimisation loops can feed in whichever they have.
5. **Do not import from [t0004]'s code** — copy the three functions we need (compute_dsi,
   compute_mean_curve, plus the angle grid generator) as small private helpers in the library, and
   keep them labelled with `# adapted from t0004/.../generate_target.py` as the single allowed
   cross-task comment.
6. **Pin library behaviour against [t0004]'s generator_params.json** — accept a
   `generator_params.json` alongside the mean CSV when computing HWHM analytically, so the test
   suite can cross-check the interpolated HWHM against the closed-form value.
7. **Keep the library pure Python** — NumPy+SciPy+pandas only. No NEURON, no matplotlib at import
   time (only inside the CLI's `--plot` flag).
